# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

React 19 + TypeScript frontend for AceReStreamer (an AceStream-to-HLS re-streamer). Built with Vite, uses bun as package manager, Chakra UI v3 for components, TanStack Router (file-based) and TanStack Query for data. The FastAPI backend lives at `../acere` (sibling directory, not covered by this file).

## Commands

- `bun run dev` — start the Vite dev server
- `bun run typecheck` — `tsc --noEmit`
- `bun run lint` — Biome check with `--write --unsafe` (auto-fixes)
- `bun run build` — typecheck + `vite build` into `dist/` (used by `docker/Dockerfile.combined`'s build stage, which copies the output into the backend image)
- `bun run build-aio` — typecheck + build, output straight into `../acere/dist` (the "all-in-one" mode where FastAPI serves the built frontend directly). This is the only production deployment — a separately hosted frontend is dev-only via `bun run dev`.
- `bun run generate-client` — regenerates `src/client/` from `./openapi.json` via `@hey-api/openapi-ts` (see `openapi-ts.config.ts`). Don't run this directly to get fresh types — use `../scripts/generate-client.sh` from the repo root inside the backend's venv, which first dumps the live OpenAPI schema from the FastAPI app into `openapi.json` and then runs this.
- `bun run generate-frontend-paths` — walks `routeTree.gen.ts` and `public/` to emit a JSON list of every frontend route/asset path, used by the backend to know which paths to hand off to the SPA in all-in-one mode.

There is no test runner configured in this package (no vitest/jest). Correctness is enforced via `typecheck` + `lint` plus manual verification in the browser.

## Architecture

**Routing**: `src/routes/` is TanStack Router file-based routing. `__root.tsx` sets up the root route (404 handling, lazy-loaded router/query devtools outside production). `_layout.tsx` is the shared authenticated-app shell (sidebar + header + `PageTitleContext` for per-page titles via `usePageTitle()`), and every page under `src/routes/_layout/*.tsx` renders inside it via `<Outlet />`. `routeTree.gen.ts` is auto-generated by the `tanstackRouter` Vite plugin — never edit it by hand.

**API client**: `src/client/` is entirely auto-generated from the backend's OpenAPI schema (see Commands above) — never hand-edit it. Services are grouped by backend router and exposed as `<Name>Service` classes (e.g. `AcePoolService`, `StreamsService`, `ScraperService`, `XtreamCodesService`). `main.tsx` points the generated client at the backend by setting `OpenAPI.BASE = baseUrl()`. `src/helpers.ts`'s `baseUrl()` resolves the API origin: `VITE_API_URL` env var if set, otherwise falls back to `window.location.origin` — this fallback is what makes all-in-one mode (frontend served by the same FastAPI process) work without any env config.

**Components**: organized by feature area under `src/components/`:
- `Admin/` — config, scraper, stream/channel and remote-config management screens, each with a `*Management.tsx` entry component plus a subdirectory of supporting pieces (e.g. `Admin/Streams/Table.tsx`)
- `Index/` — the status/dashboard page (AcePool instance table, backend info)
- `info/iptv/` — IPTV/Xtream Codes connection info shown to end users
- `Common/` — app chrome (Header, Sidebar, NotFound)
- `ui/` — thin wrappers around Chakra UI v3 primitives (button, dialog, table, code, etc.) — prefer these over importing straight from `@chakra-ui/react` when a wrapper already exists here, since they carry project-specific defaults (e.g. `ui/code.tsx` sets the monospace font).

**State/data fetching**: TanStack Query for all server state (`useQuery`/`useMutation`, no global client-side store). Toasts via `useCustomToast`, form/API error formatting via `handleError` in `src/utils.ts`.

**Path alias**: `@/*` maps to `src/*` (configured in both `tsconfig.json` and `vite.config.ts`).

**Chakra UI skills**: `.agents/skills/chakra-ui-builder`, `chakra-ui-refactor`, `chakra-ui-migrate` contain reference material for building/refactoring/migrating Chakra v3 components — check there for theming and component-choice guidance before introducing new UI patterns.

Note: the top-level repo `README.md` mentions EPG support and user/login management — these have been removed from the app (no auth or EPG code exists in this frontend); don't rely on the README for current feature scope.
